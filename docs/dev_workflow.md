# DEV_WORKFLOW.md

## Purpose

This document defines the development rules and architecture constraints for AI coding agents (Cursor, Antigravity, Copilot, etc.) working on the Squeegee Samurai project.

Agents should treat this document as **authoritative guidance** for how the application is designed and what components are allowed.

The system intentionally uses a **stateless architecture with no database**.

---

# Core Architecture

The application follows a simple flow:

Browser Quote Calculator  
→ Vercel Serverless Function  
→ Email Notifications

There is **no persistent backend storage**.

The email inbox becomes the **system of record** for estimate requests.

---

# Removed Systems (Do Not Reintroduce)

The following components were intentionally removed from the architecture:

| System | Status |
|------|------|
| Supabase | Removed |
| Postgres | Removed |
| Database ORM | Removed |
| PDF generation | Removed |
| File storage | Removed |
| Signed URLs | Removed |

Agents must **not reintroduce these systems** unless explicitly instructed.

# Quote Flow

1. Customer fills out estimate form
2. Estimate is calculated **client-side**
3. Customer submits contact information
4. Browser sends payload to serverless endpoint
5. Endpoint sends emails
6. Business owner manually enters data into CRM

---

# API Endpoint

```
POST /api/submit-estimate
```

The endpoint must remain **stateless**.

No data should be written to any database or storage system.

---

# Expected Request Payload

Example:

```json
{
  "customer": {
    "name": "Jane Doe",
    "email": "jane@email.com",
    "phone": "555-555-5555"
  },
  "property": {
    "address": "123 Main St",
    "type": "residential"
  },
  "estimate": {
    "line_items": [],
    "subtotal": 150,
    "uplift": 20,
    "total": 170
  },
  "notes": "Optional notes",
  "timestamp": 1712345678
}
```

The payload is generated by the frontend calculator.

---

# Serverless Function Responsibilities

The serverless endpoint must:

1. Validate required fields
2. Sanitize inputs
3. Format email content
4. Send email to the business owner
5. Send confirmation email to the customer

The function **must not store data anywhere**.

---

# Email Behavior

Two emails are sent.

## Owner Email

Recipient:

OWNER_EMAIL

Includes:

* customer name
* email
* phone
* property address
* estimate breakdown
* subtotal
* uplift
* total estimate
* notes
* submission timestamp

Subject example:

New Estimate Request — Jane Doe — $170

---

## Customer Confirmation Email

Recipient:

customer.email

Includes:

* thank you message
* estimate summary
* business contact information
* disclaimer that final pricing may vary

Subject example:

Your Squeegee Samurai Estimate

---

# Environment Variables

Required environment variables in Vercel:

```
RESEND_API_KEY
OWNER_EMAIL
APP_ENV
```

Optional:

```
ENABLE_CUSTOMER_CONFIRMATION=true
```

Agents should never hardcode secrets.

---

# Validation Rules

Serverless endpoint must validate:

Required:

* customer.name
* customer.email
* property.address
* estimate.total

Optional:

* phone
* notes

Invalid payloads must return an error.

---

# Error Handling

If the owner email fails:

* return HTTP 500
* frontend should allow retry

Errors should be logged using Vercel runtime logs.

---

# Spam Protection (Recommended)

Frontend should implement one of:

* Cloudflare Turnstile
* Google reCAPTCHA

---

# Agent Development Rules

Agents should:

* maintain stateless architecture
* keep serverless functions lightweight
* follow payload contract
* avoid unnecessary dependencies

Agents must not:

* add databases
* introduce ORM libraries
* add background job systems
* generate files
* add storage layers
